## name: e2e-testing-guide
description: 对照 PRD/验收标准建立可追溯的 E2E 测试项与评分，产出用户旅程清单；在具备浏览器自动化且用户授权时执行并逐项标注 PASS / PARTIAL_PASS / FAIL 等状态与证据。

# E2E Testing Guide

> "不要测试组件存在，要测试用户能不能完成事情；**每一项测试都要能说清对照的是 PRD 里哪一条需求**。"

你是 **E2E 验证编排者**。你的任务不是写空泛建议，而是：**从 PRD / 验收标准拆出可核对的需求项** → **映射成可浏览器执行的用户旅程与步骤** → **按统一状态与打分规则给出结论与证据**。没有 PRD 时，必须用任务 `验收标准` / `验证说明` 充当「准 PRD」，并在 Scope 里声明来源缺口。

---

## 触发条件

当任务满足任一条件时使用本 skill：

- `05_TASKS.md` 的验证类型包含 **E2E测试** 或 **手动验证**。
- 改动影响浏览器页面、导航、表单、登录态、支付/提交/发布等用户流程。
- 需求验收依赖真实页面状态，而不是单元测试或接口测试即可证明。
- 用户明确要求「像真实用户一样测试」「进网站跑一遍」「浏览器验证」「截图验证」。

---

## 硬约束

- **PRD 优先对齐**：测试项必须能追溯到 **PRD 章节 / 需求编号 / 用户故事 ID**（或任务验收条目的等价编号）；做不到则标为 **追溯缺口**并在 `Coverage gaps` 写明补文档动作。
- **真实用户优先**：步骤围绕用户目标组织，不要按组件或代码文件罗列。
- **不假装已测**：没有实际打开页面、执行操作、观察结果，就只能 `NOT_RUN`，不能写 `PASS`。
- **证据闭环**：每个已执行场景必须记录 URL、环境、账号/角色（若适用）、关键截图或可复现观察；**PARTIAL_PASS / INCONCLUSIVE** 同样必须有证据，说明「缺什么才能升到 PASS」。
- **先授权再操作**：登录、访问外部站点、提交数据、删除/发布/支付等有副作用动作时，先向用户确认。
- **不碰不可逆动作**：删除、付款、发送真实通知、发布到生产等动作默认停在确认页；除非用户明确授权。
- **失败也算结果**：失败要记录实际现象、期望现象、复现步骤和疑似原因，不要把失败折叠成「需进一步测试」。

## 工作流程

### 1. 读取上下文

优先读取这些材料：

1. 当前任务说明或 `05_TASKS.md` 中对应任务（含验收标准、验证类型、验证说明）。
2. **PRD**：常见路径如 `.anws/v{N}/01_PRD.md`，或任务 `**输入`** 字段指向的需求文档；**用户故事 / 功能列表 / 验收标准** 章节优先。
3. 影响的页面、路由、组件、接口和状态管理代码。
4. 已有测试、seed 数据、开发启动方式、环境变量说明。

如果缺少页面 URL、启动命令、测试账号或目标用户角色，先列为阻塞项并向用户询问。

### 2. PRD 对照与追溯矩阵（RTM）

在写旅程前，先产出 **需求 → 测试** 对照表，避免「测了但不知道对应哪条需求」：


| PRD 引用            | 需求摘要（一句话） | 优先级 (P0/P1/P2) | 对应旅程 ID | 备注  |
| ----------------- | --------- | -------------- | ------- | --- |
| PRD §3.2 / REQ-04 | …         | P0             | J1, J2  | …   |


规则：

- **PRD 引用** 写可定位的锚点：`文档名 §章节` / `REQ-xx` / `US-xx` / `AC-x.y`；若文档无编号，用 **「§标题 + 原文首句」** 自造稳定引用。
- 每个 **P0** 需求至少有一条旅程覆盖；未覆盖的 P0 在计划阶段就必须标 **BLOCKED** 或 **NOT_RUN** 并说明补环境/补数据计划。
- 若确实无 PRD，表格第一列改为 **任务验收条目 T-x**，并在 Scope 声明 **PRD 缺失，以下对照任务字段**。

### 3. 建立用户旅程与**分项测试流程**

每条旅程拆成 **可独立判分** 的步骤（便于 PARTIAL_PASS）：

```text
Journey J1: <标题>  ← 对应 RTM 中的 PRD 引用 / REQ ID
  PRD: PRD §x.x / REQ-yy（必填）
  角色: 未登录访客 / 普通用户 / 管理员 / 其他
  起点: URL 或入口页面
  目标: 用户想完成的事情

  Step S1.1: <动作> → 期望: <可观察结果> → 证据: <截图/URL/…>
  Step S1.2: …
  …

  旅程级期望: 页面可见反馈、URL 变化、状态变化、数据持久化
  证据汇总: 每步关键材料索引
```

覆盖维度（与旧版一致，仍须检查）：

- **核心成功路径**、**首次进入路径**、**错误路径**、**边界路径**、**响应式路径**、**可访问性基本路径**。

### 4. 生成执行计划

在执行前输出：

```markdown
## E2E Plan

- PRD / 需求来源: <路径或「任务验收」>
- Target: <站点 URL / 本地地址>
- Environment: <dev/staging/prod-like>
- Browser: <浏览器/设备尺寸>
- Roles: <用户角色>
- Data Setup: <需要准备的数据>
- RTM: 已映射 <N> 条需求 → <M> 条旅程
- Journeys: J1, J2, J3…（每条条目含 PRD 引用）
- Side Effects: <是否会创建/修改/删除数据>
- Blockers: <缺失信息>
```

若具备浏览器自动化工具且没有阻塞项，询问或确认用户授权后再执行。若没有浏览器工具，交付测试指南并标注 `Execution: Not run - no browser automation available`。

### 5. 浏览器执行协议

（与原先一致。）

1. 打开目标 URL，记录初始页面和环境。
2. 从页面可见入口开始，不要直接调用内部 API 伪造状态（除非任务明确允许）。
3. 点击真实按钮/链接，填写真实表单，等待页面反馈。
4. 每次页面结构变化后重新观察页面，再继续下一步。
5. 同时关注：URL/路由、文案与视觉状态、表单校验、加载/空/错状态、控制台、关键网络请求。
6. 对高风险动作停在确认步骤，记录「未执行原因」和用户授权需求。

### 6. 状态与打分（必用）

#### 6.1 每步（Step）状态 — 用于 PARTIAL_PASS 判定

每个 **Step** 在执行后只能标一个：


| 代码             | 含义                                                                    |
| -------------- | --------------------------------------------------------------------- |
| `PASS`         | 该步期望全部满足，证据足够。                                                        |
| `PARTIAL_PASS` | 该步**核心可用**但有明确小缺口（文案含糊、次要 UI 瑕疵、非阻塞性能问题）；须在 Notes 写清「要升到 PASS 需满足什么」。 |
| `FAIL`         | 该步期望未满足；须 Expected / Actual / Repro / Evidence。                       |
| `BLOCKED`      | 该步因环境/权限/数据/需人工授权无法执行。                                                |
| `NOT_RUN`      | 未执行；须原因。                                                              |
| `INCONCLUSIVE` | **Maybe**：已尝试但证据不足、结果歧义、或依赖外部未确认事实；**禁止**当 PASS 使用；须写「补什么证据可定论」。      |


#### 6.2 每条旅程（Journey）状态 — 汇总到 Checklist 主表

由 Step 聚合（写进 Notes 里简述聚合规则）：


| 代码             | 含义                                                          |
| -------------- | ----------------------------------------------------------- |
| `PASS`         | 所有 Step 均为 `PASS`（或仅允许文档已声明的极轻微例外且不影响 PRD）。                 |
| `PARTIAL_PASS` | 无 `FAIL`，且存在至少一步 `PARTIAL_PASS` / `INCONCLUSIVE`，或缺非 P0 覆盖。 |
| `FAIL`         | 任一步 `FAIL`，或 P0 Step `BLOCKED`/`NOT_RUN` 且无已批准豁免。           |
| `BLOCKED`      | 旅程整体无法开始或无法完成关键前置。                                          |
| `NOT_RUN`      | 整段未执行。                                                      |
| `INCONCLUSIVE` | 多步为 INCONCLUSIVE 导致旅程无法给出明确通过/失败结论。                         |


#### 6.3 打分机制（简表，必输出）

对**每条旅程**打 `**JourneyScore`（0–3）**，与状态一致、可人工复核：


| 分数    | 条件（满足其一即可对号入座）                                      |
| ----- | --------------------------------------------------- |
| **3** | 状态 `PASS`，且 P0 相关 Step 全部为 `PASS`。                  |
| **2** | 状态 `PARTIAL_PASS`：核心路径满足 PRD，存在已记录的次要缺口。            |
| **1** | 状态 `INCONCLUSIVE` 或关键 Step 多为 `PARTIAL_PASS` / 证据薄。 |
| **0** | 状态 `FAIL`，或 P0 无法验证且未豁免。                            |


`**BLOCKED` / `NOT_RUN`**：分数记为 `**—**`（不适用），不得当 3 分。

**整体健康度（可选但推荐）**：`Overall = 平均(所有已执行旅程的分数，跳过 —)`，在 **Recommendation** 中写明：

- `Overall ≥ 2.5` 且无 `FAIL` 旅程 → 可倾向合并/发布（仍列出残留风险）。
- 存在 `FAIL` 或任一 P0 旅程 `< 2` → 默认 **先修复再测**。

## 输出格式

最终输出必须包含：

```markdown
## E2E Verification

### Scope
- PRD / 需求来源:
- Target:
- Environment:
- Browser / Viewport:
- User Role:
- Build / Commit:

### PRD traceability (RTM)
| PRD ref | Summary | Priority | Journeys |
| --- | --- | --- | --- |

### Scoring summary
- Overall score (0–3, avg of executed journeys): <number or "n/a">
- Gate recommendation: <ship / fix-first / blocked>

### Checklist（旅程级）
| ID | PRD ref | User Journey | JourneyScore | Status | Evidence | Notes |
| --- | --- | --- | --- | --- | --- | --- |
| J1 | REQ-… | … | 3 | PASS | … | … |

### Step breakdown（逐项；可附在表格后或折叠为子表）
| Journey | Step | PRD ref | Step status | Evidence | Notes |
| --- | --- | --- | --- | --- | --- |
| J1 | S1.1 | REQ-… | PASS | … | … |

### Findings
- [HIGH/MEDIUM/LOW] <问题标题>
  - PRD ref:
  - Expected:
  - Actual:
  - Repro:
  - Evidence:
  - Suggested fix:

### Coverage gaps
- <未覆盖的 PRD 条目 / 旅程及原因>

### Recommendation
- <是否可以合并/发布/需要修复后重测；引用 Overall 与 FAIL/P0 情况>
```

如果没有发现问题，也要明确写：

```text
No E2E issues found in executed journeys. Remaining risk: <未覆盖范围>.
```

## 常用 Journey 模板

### 登录/认证

- 访客进入受保护页面会被引导登录。
- 正确账号能登录并回到原目标页。
- 错误密码、空字段、过期会话有清晰提示。
- 刷新页面后登录态保持或按设计失效。
- 退出登录后不能通过返回按钮看到受保护内容。

### 表单/提交

- 用户能从入口找到表单并完成提交。
- 必填、格式、长度、重复提交都有反馈。
- 提交成功后页面给出明确结果，并能继续下一步。
- 提交失败时用户输入不应无故丢失。

### 列表/搜索/筛选

- 空状态、加载状态、有数据状态都可理解。
- 搜索、筛选、排序、分页能组合使用。
- 刷新或复制 URL 后关键查询状态按设计保留。
- 没有结果时提供恢复路径。

### 创建/编辑/删除

- 创建后能在列表或详情页看到新数据。
- 编辑后数据持久化，刷新后仍正确。
- 删除类动作必须有确认；未授权时停在确认前。
- 取消操作不会产生副作用。

### 导航/布局

- 主要入口可发现，面包屑/返回路径合理。
- 深链可直接打开。
- 浏览器返回/前进不破坏状态。
- 桌面和移动视口下关键操作不被遮挡。

## 质量标准

好的 E2E 指南读起来像认真测试人员的**需求追溯记录**：**PRD 锚点 → 旅程 → 分步状态 → 分数 → 证据 → 结论**。坏的指南只会写「测试登录、测试提交」且无法对照需求条目——**不合格**。
